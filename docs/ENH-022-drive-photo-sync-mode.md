# ENH-022: Google Drive 사진 동기화 (업로드/다운로드 단일 모드)

## 목표
텍스트 이벤트 공유 기능과 분리된 **사진 전용 Google Drive 동기화**를 도입합니다.

관련 논의 요약:

- 텍스트 동기화(Export/Import)와 사진 동기화를 분리
- 앱 1개 인스턴스는 Drive 계정/폴더 1개만 연결
- 동기화 모드는 `업로드` 또는 `다운로드` 중 하나만 선택
- 기본 모드는 `다운로드`
- 폴더 입력은 사용자가 복붙하는 Drive 웹 링크(URL)
- 삭제 전파는 양쪽 모두 반영

핵심 목표:

- 설정 다이얼로그에 `드라이브 동기화` 탭을 추가합니다.
- 같은 폴더를 바라보는 업로드 폰/다운로드 폰 간에 사진 목록을 동기화합니다.
- 업로드 시 WebP 인코딩(중/고 품질)으로 용량을 절감합니다.
- 업로드 폰에서 삭제한 사진은 Drive에서도 삭제하고, 다운로드 폰에도 삭제를 반영합니다.
- Drive 폴더 URL이 바뀌면 기존 동기화 상태를 안전하게 리셋합니다.

---

## 비목표 (이번 범위에서 제외)

- 텍스트 이벤트 동기화 정책 변경
- 한 앱이 여러 Drive 계정/폴더를 동시에 관리
- 한 앱에서 업로드/다운로드 동시 활성화
- 다중 업로드 작성자 충돌 해결(두 폰이 같은 Drive에 동시에 업로드)
- 실시간 푸시(FCM) 기반 동기화 알림

---

## 현재 상태와 갭

현재 앱은 다음 특성을 가집니다.

- 텍스트 이벤트 Export/Import는 `PHOTO`를 명시적으로 제외합니다.
- 사진 동기화는 로컬 `MediaStore` 스캔 기반이며 Drive 연동이 없습니다.
- 사진 레코드는 `localUri` 중심이라 디바이스 간 공유 식별자가 없습니다.

따라서 Drive 동기화를 위해서는 **원격 식별자와 동기화 출처 메타데이터**가 필요합니다.

---

## 사용자 시나리오

### 1) 설정 진입

옵션 다이얼로그에 `드라이브 동기화` 탭을 추가합니다.

탭 구성:

1. Drive 폴더 주소 입력창 (웹 링크)
2. 동기화 모드 선택 (라디오):
   - 다운로드 (기본)
   - 업로드
3. 저장/적용 버튼
4. 최근 동기화 결과 요약 (시간, 추가/삭제/실패 건수)

### 2) 다운로드 모드

- URL 저장 직후 즉시 1회 동기화 실행
- 이후 주기 동기화(예: 6시간 간격)
- Drive 신규 파일은 로컬에 반영
- Drive에서 삭제된 파일은 앱에서도 삭제(또는 soft-delete 숨김)

### 3) 업로드 모드

- URL 저장 후 업로드 모드로 전환 시 인증 필요(OAuth)
- 인증 완료 후 즉시 1회 동기화 실행
- 이후 주기 동기화
- 로컬 신규 동기화 대상은 Drive로 업로드
- 삭제 케이스 전파:
  - 케이스 A: 폰 원본 파일 삭제
  - 케이스 B: 앱에서 사진 이벤트만 삭제(원본은 남음)
  - 두 케이스 모두 Drive 파일 삭제로 전파

### 4) 업로드/다운로드 단말 역할 분리

- **업로드 단말**은 로컬 원본(MediaStore/로컬 URI)을 기준으로 화면에 표시합니다.
- 업로드 단말은 Drive 업로드를 수행하지만, 표시를 위해 Drive 파일을 재다운로드하지 않습니다.
- **다운로드 단말**은 Drive 동기화 자산을 기준으로 표시합니다.
- 다운로드 단말에서는 로컬 사진과 동기화 사진이 혼재될 수 있으며, `syncOrigin`에 따라 로딩 경로를 분기합니다.

### 5) 폴더 URL 변경

URL 변경 시 기존 동기화 상태를 전부 리셋합니다.

단, 리셋 원칙:

- 동기화 출처 메타데이터/매핑/커서/캐시만 리셋
- 동기화와 무관한 기존 로컬 사진 이벤트는 보존

## 데이터 모델 변경안

기존 `PhotoRecord.localUri` 단독 구조를 확장합니다.

권장 필드(예시):

```kotlin
enum class PhotoSyncOrigin {
    LOCAL,          // 기존 로컬 파이프라인
    DRIVE_UPLOAD,   // 업로드 모드에서 생성
    DRIVE_DOWNLOAD  // 다운로드 모드에서 수신
}

data class PhotoRecord(
    ...,
    val localUri: String,
    val syncOrigin: PhotoSyncOrigin = PhotoSyncOrigin.LOCAL,
    val driveFileId: String? = null,          // full 이미지 fileId
    val driveThumbnailFileId: String? = null, // thumbnail 이미지 fileId
    val driveFolderId: String? = null,
    val contentHash: String? = null,
    val encodedFormat: String? = null,      // e.g. "webp"
    val encodedQualityTier: String? = null, // e.g. "medium" | "high"
    val syncDeletedAt: Long = 0L            // soft-delete 추적용
)
```

보조 테이블(권장):

- `DriveSyncState`
  - currentMode(upload/download)
  - linkedAccountId
  - linkedFolderUrl
  - linkedFolderId
  - lastSyncAt
  - syncCursor/pageToken
- `DriveSyncTombstone`
  - driveFileId / contentHash
  - deletedAt
  - sourceDeviceId

`Tombstone`은 삭제 전파 안정성을 높이는 선택지입니다. 단일 작성자 모델에서는 필수는 아니지만, 재동기화/재다운로드 방지에 유리합니다.

## Drive 링크 처리

사용자 입력은 URL 문자열을 받습니다.

지원 예시:

- `https://drive.google.com/drive/folders/{folderId}`
- `https://drive.google.com/drive/u/0/folders/{folderId}`
- 공유 링크 변형 URL

정규화 단계:

1. URL에서 `folderId` 추출
2. 내부 상태에는 `folderId`를 canonical 값으로 저장
3. UI에는 원본 URL/정규화 성공 여부를 표시

유효성 실패 시:

- 저장 차단
- "폴더 링크를 확인해주세요" 메시지 표시

## 동기화 파이프라인

## 공통

- WorkManager 기반 주기 실행
- 네트워크 가능 시 동작
- 실패 시 지수 백오프
- 수동 "지금 동기화" 버튼으로 즉시 1회 실행 가능

### 썸네일/원본 이중 자산 전략

동기화 자산은 `thumbnail`과 `full` 두 종류를 분리 저장합니다.

- 업로드 시: 각 사진마다 `thumbnail(리스트/그리드용)` + `full(상세/확대용)` 2개 생성 및 업로드
- 다운로드 시: 기본적으로 `thumbnail`을 먼저 동기화
- 상세/확대 진입 시: `full`이 로컬에 없으면 지연 다운로드(on-demand)
- `full` 다운로드 완료 후에는 캐시에 보관하며, 캐시 정책에 따라 정리 가능

권장 규격:

- thumbnail: 긴 변 480~720px
- full: 긴 변 1920px (WebP lossy)

### 다운로드 모드 파이프라인

1. Drive 폴더 목록 조회
2. 기존 매핑(`driveFileId`)과 비교
3. 신규 파일 다운로드 및 로컬 저장
4. 앱 DB 반영 (`syncOrigin = DRIVE_DOWNLOAD`)
5. 누락 파일(이전엔 있었는데 현재 폴더에서 사라짐) 처리
   - 기본: 앱에서도 삭제(soft-delete 후 정리)
6. 다운로드 파일 로컬 캐시 저장(앱 전용 저장소)
   - 원격 직접 스트리밍이 아니라, 오프라인/성능/안정성을 위해 로컬 캐시에 저장 후 표시
7. 동기화 요약 저장 및 알림

### 업로드 모드 파이프라인

1. 로컬에서 동기화 대상 추출
2. WebP 인코딩(중/고 품질)
3. 해시 계산 후 중복 확인
4. Drive 업로드
5. DB에 `driveFileId`/해시/인코딩 메타 저장
6. 삭제 큐 처리
   - 케이스 A/B 모두 Drive 삭제 API 호출
7. 동기화 요약 저장 및 알림

## 파일 용량/캐시 정책

### 해상도/인코딩 확정

- 긴 변 기준 최대 해상도 캡: **1920px 고정**
- 포맷: `WebP lossy`
- 품질 티어: `medium/high` 내부 운용 가능하나, 릴리즈 기본값은 `high`로 시작

선정 이유:

- 2560px 대비 파일 용량과 디코딩 비용을 줄여 타임라인/상세 화면 체감 지연 가능성을 낮춥니다.
- 공유/백업 목적의 사진 조회에서는 1920px가 대부분의 스마트폰 화면에서 충분한 품질을 제공합니다.

### 다운로드 저장 방식

- 다운로드 모드는 Drive 원격 파일을 직접 스트리밍하지 않고, **앱 전용 로컬 캐시에 저장 후 표시**합니다.
- 목적: 오프라인 조회, 스크롤 성능 안정화, 네트워크 변동 내성 확보.

### 캐시 유지/정리 정책

- 로컬 저장은 "영구 보관"이 아니라 **캐시 정책 기반 유지**입니다.
### 뷰포트 기반 로딩 정책 (리스트/그리드)

- 리스트/그리드는 **viewport 기반 lazy loading**을 사용합니다.
- 스크롤 중에는 **썸네일 우선 로딩**, 원본 해상도는 상세 진입 시 로딩합니다.
- 프리페치 범위는 화면 앞뒤 `N`개(초기 권장: 6~12개)로 제한합니다.
- 메모리 압박 신호가 오면 썸네일 캐시도 단계적으로 축출합니다.
- `syncOrigin == LOCAL`은 기존 로컬 URI 로딩 경로를 사용합니다.
- `syncOrigin == DRIVE_DOWNLOAD`은 thumbnail 우선 로딩 후, 상세/확대에서 full on-demand 로딩을 사용합니다.

- 기본 정책:
  1. 최근 조회 우선(LRU) 유지
  2. 동기화 삭제 반영 시 해당 캐시 즉시 정리
  3. 앱 저장공간 압박 시 오래된 원본 캐시부터 정리
- 썸네일 캐시는 원본보다 오래 유지해 UX를 보존합니다.

### 예상 용량 가이드(1920px, WebP)

이미지 내용에 따라 편차가 크지만, 운영 추정치는 다음과 같습니다.

- 1장: 약 **0.25MB ~ 1.2MB**
- 1,000장: 약 **0.25GB ~ 1.2GB**
- 3,000장: 약 **0.75GB ~ 3.6GB**

주의: 노이즈가 많거나 디테일이 많은 사진은 상한을 초과할 수 있습니다.

## 사진 인코딩 정책 (WebP)

- 포맷: `WebP lossy`
- 티어:
  - `중`: 품질 75~82 범위
  - `고`: 품질 88~92 범위

보완 정책:

- 긴 변 기준 최대 해상도 캡: **1920px 고정**
- EXIF/메타데이터 최소화 저장(개인정보 노출 절감)
- 원본 보존 필요 시 로컬 원본은 손대지 않고 업로드본만 인코딩

주의:

- "항상 WebP가 절대 최적"은 아님
- 그러나 Android 호환성과 용량/화질 균형 측면에서 v1 기본값으로 적합

## 삭제 정책 상세

삭제 반영 기준은 **동기화 대상(PhotoRecord with drive linkage)** 입니다.

### 업로드 모드

- 원본 파일 삭제 감지 또는 앱 이벤트 삭제 시 삭제 큐 적재
- 큐 처리 워커가 Drive 파일 삭제
- 성공 시 로컬 레코드 `syncDeletedAt` 갱신(또는 hard-delete)

### 다운로드 모드

- 폴더 목록 diff에서 사라진 `driveFileId` 감지
- 앱에서도 해당 항목 삭제/숨김 반영
- 사용자에게 요약 알림(삭제 n건)

정책 제안:

- UI 표시: 기본은 hard-delete처럼 보이되, 내부적으로는 soft-delete 후 정리 가능
- 안전장치: 대량 삭제 **50건 이상** 감지 시 1회 확인 배너

## 알림(노티) 정책

실시간 푸시 대신 **동기화 완료 요약 알림**으로 시작합니다.

알림 예시:

- "드라이브 동기화 완료: 추가 12, 삭제 3"
- 실패 시: "드라이브 동기화 일부 실패: 재시도 예정"

앱 내 최근 동기화 로그를 함께 제공해 알림을 놓쳐도 상태 확인 가능하게 합니다.

## 인증/권한

### 다운로드 모드

- 링크/권한이 열려 있다면 계정 로그인 없이 read 가능(범위/링크 정책에 따름)

### 업로드 모드

- Google 로그인 + Drive 파일/폴더 접근 권한(OAuth) 필요

오류 메시지 예시:

- "업로드 모드는 Google 계정 연결이 필요합니다"
- "폴더 접근 권한이 없습니다. 공유 설정을 확인해주세요"

## 상태 리셋 규칙 (폴더 URL 변경)

URL 변경 시 반드시 수행:

1. 동기화 커서/pageToken 초기화
2. 폴더 매핑(`driveFileId`, `driveFolderId`) 초기화
3. 동기화 캐시/큐 초기화
4. 기존 동기화 산출물 정리(정책 선택):
   - A안: `DRIVE_DOWNLOAD` 출처 데이터만 제거
   - B안: 모든 Drive-linked 데이터 제거

권장: A안부터 시작하여 사용자 놀람을 줄임.

## 가능성/적합성 검토

### 가능한가?
가능합니다. 현재 앱이 WorkManager 기반 백그라운드 동작과 사진 파이프라인을 이미 보유하고 있어 구조적으로 확장 가능합니다.

### 문서 적합성
본 문서는 제품 요구사항(모드, URL 입력, 삭제 전파, 리셋 규칙)을 구현 가능한 단위(데이터 모델, 워커 파이프라인, 단계별 출시)로 분해했습니다.

### 확정된 결정사항

## 수용 기준 (Acceptance Criteria)

1. 설정에서 Drive 폴더 URL + 모드(업로드/다운로드)를 저장할 수 있다.
2. 기본 모드는 다운로드이며, 저장 시 즉시 1회 동기화가 실행된다.
3. 업로드 모드 전환 시 인증이 없으면 업로드가 시작되지 않는다.
4. 업로드 모드에서 삭제 케이스 A/B가 Drive 삭제로 전파된다.
5. 다운로드 모드에서 Drive 삭제가 앱 삭제/숨김으로 반영된다.
6. 폴더 URL 변경 시 동기화 상태는 리셋되며, 비동기화 로컬 데이터는 보존된다.
7. 다운로드 파일은 앱 전용 로컬 캐시에 저장되어 오프라인에서도 최근 동기화분을 조회할 수 있다(영구보관이 아닌 캐시 정책 기반).
8. 대량 삭제 50건 이상 감지 시 사용자 확인/경고 흐름이 동작한다.
9. 동기화 후 사용자에게 추가/삭제/실패 요약을 확인할 수 있다(알림 또는 앱 내 로그).
