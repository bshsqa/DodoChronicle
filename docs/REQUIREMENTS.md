## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 앱 이름 | DodoChronicle |
| 플랫폼 | Android |
| 주요 사용자 | 자녀의 성장을 기록하고 싶은 부모 |
| 핵심 가치 | 카카오톡 대화와 사진을 자동 수집하여 아이의 성장 타임라인을 시각화 |

### 핵심 사용 시나리오

1. 처음 앱 설치 → 초기화 마법사 실행 → 사진 전체 분류 → 아이 그룹 선택 → 카카오톡 파일 import → 타임라인 완성
2. 이후 앱 실행마다 신규 사진 자동 분류 + 카카오톡 추가 import
3. 타임라인을 핀치줌으로 탐색, 이벤트 열람·관리

## 2. 시스템 아키텍처

```
┌───────────────────────────────────────────────────────┐
│                    Presentation Layer                 │
│  TimelineScreen │ InitWizardScreen │ EventDetailScreen│
│  (Jetpack Compose + MotionLayout)                     │
├───────────────────────────────────────────────────────┤
│                    ViewModel Layer                    │
│  TimelineViewModel │ InitViewModel │ PhotoScanViewModel│
│  (StateFlow + Hilt)                                   │
├──────────────┬──────────────┬───────────────────────── ┤
│  Use Cases   │              │                          │
│  ImportKakao │  ScanPhotos  │  ClassifyEvent           │
│  SyncNewPhoto│  ManageEvent │  FaceMatch               │
├──────────────┴──────────────┴──────────────────────────┤
│                    Data Layer                          │
│  Room DB  │  MediaStore  │  FileSystem (KakaoTalk .txt)│
├───────────────────────────────────────────────────────┤
│                    ML / AI Layer                       │
│  ML Kit Face Detection │ TFLite FaceNet (임베딩)       │
│  Claude / Gemini API (이벤트 분류, 온라인)              │
├───────────────────────────────────────────────────────┤
│                    Background Layer                    │
│  WorkManager (신규 사진 스캔) │ ContentObserver (갤러리)│
└───────────────────────────────────────────────────────┘
```

### 아키텍처 패턴
- **Clean Architecture** (Presentation / Domain / Data 3계층)
- **MVVM** + Unidirectional Data Flow (UDF)
- **Single Activity** + Compose Navigation

## 3. 기능 요구사항

### 3.1 앱 상태 머신

앱은 세 가지 상태를 가지며, 상태에 따라 동작이 달라진다.

```
         최초 설치
             │
             ▼
    ┌─────────────────┐
    │  INITIALIZING   │  ← 전체 사진 스캔 + 얼굴 클러스터링 + 사용자 그룹 선택
    └────────┬────────┘
             │ 그룹 선택 완료
             ▼
    ┌─────────────────┐
    │    ACTIVE       │  ← 타임라인 탐색, 이벤트 관리, 카카오톡 import
    └────────┬────────┘
             │ 앱 재시작
             ▼
    ┌─────────────────┐
    │   SYNC (백그라운드)│  ← 신규 사진만 스캔, 기존 임베딩 재활용
    └─────────────────┘
             │ 완료 (불확실한 사진은 사용자에게 확인)
             ▼
           ACTIVE
```

#### 3.1.1 INITIALIZING 상태

| ID | 요구사항 |
|---|---|
| INIT-01 | 첫 실행 시 자동으로 초기화 마법사(Wizard)를 시작한다 |
| INIT-02 | 기기 내 전체 사진을 대상으로 얼굴 탐지(ML Kit)를 수행한다 |
| INIT-03 | 탐지된 얼굴을 임베딩(TFLite FaceNet) 기반 유사도로 클러스터링한다 |
| INIT-04 | 클러스터 결과를 썸네일 그리드 UI로 사용자에게 보여준다 |
| INIT-05 | 사용자가 하나 이상의 클러스터를 선택하면 해당 클러스터들이 대상 아이로 확정된다 |
| INIT-06 | 확정된 클러스터의 평균 임베딩 벡터를 DB에 저장한다 |
| INIT-07 | 설정 화면에서 초기화를 재실행할 수 있다 |

#### 3.1.2 ACTIVE 상태

| ID | 요구사항 |
|---|---|
| ACT-01 | 타임라인 탐색, 이벤트 열람, 사진 열람이 가능하다 |
| ACT-02 | 카카오톡 내보내기 파일(.txt)을 import할 수 있다 |
| ACT-03 | 잘못 분류된 사진을 타임라인에서 제거할 수 있다 |
| ACT-04 | 잘못 파싱된 이벤트를 삭제할 수 있다 |
| ACT-05 | 수동으로 이벤트를 추가할 수 있다 |
| ACT-06 | 이벤트에 즐겨찾기(★)를 설정하고, 즐겨찾기 필터로 조회할 수 있다 |

#### 3.1.3 SYNC 상태 (백그라운드)

| ID | 요구사항 |
|---|---|
| SYNC-01 | 앱 재시작 시 마지막 스캔 이후 추가된 신규 사진만 대상으로 스캔한다 |
| SYNC-02 | 기존 저장된 얼굴 임베딩과 코사인 유사도를 비교하여 자동 분류한다 |
| SYNC-03 | 유사도가 상단 임계값(예: 0.75) 이상이면 자동으로 타임라인에 추가한다 |
| SYNC-04 | 유사도가 상단 임계값 미만이고 하단 임계값(예: 0.4) 이상인 경우(애매한 사진)에만 사용자에게 확인 요청 알림을 표시한다 |
| SYNC-04b | 유사도가 하단 임계값(예: 0.4) 미만이면 아이와 무관한 사진으로 판단하여 사용자에게 확인 없이 자동으로 제외한다 |
| SYNC-05 | 사용자가 확인 요청에서 수락/거부를 선택하면 결과를 DB에 반영한다 |
| SYNC-06 | WorkManager로 스캔을 스케줄링하며, ContentObserver로 갤러리 변경을 감지한다 |

### 3.2 카카오톡 대화 파싱

> 카카오톡은 외부 API를 제공하지 않으므로, "대화 내보내기" 기능으로 생성된 `.txt` 파일을 import한다.

| ID | 요구사항 |
|---|---|
| KAKO-01 | 사용자가 `.txt` 파일(단일 또는 복수)을 선택하면 파싱을 시작한다 |
| KAKO-02 | 파싱된 메시지는 발신자, 날짜/시각, 내용으로 구조화하여 DB에 저장한다 |
| KAKO-03 | 동일 방의 파일을 재import할 경우, 마지막 저장 타임스탬프 이후 내용만 추가한다 (증분 업데이트) |
| KAKO-04 | 중복 메시지는 날짜+내용 해시(SHA-256)로 감지하여 skip한다 |
| KAKO-05 | AI(Claude/Gemini API)를 사용하여 메시지에서 아이 관련 이벤트를 추출한다 |
| KAKO-06 | 여러 채팅방 파일을 각각 import하여 통합 관리할 수 있다 |

### 3.3 사진 수집 및 얼굴 인식

| ID | 요구사항 |
|---|---|
| PHOTO-01 | Android MediaStore API를 통해 기기 내 사진에 접근한다 |
| PHOTO-02 | ML Kit Face Detector로 각 사진에서 얼굴 영역을 탐지한다 |
| PHOTO-03 | TFLite FaceNet 모델로 탐지된 얼굴의 128차원 임베딩 벡터를 추출한다 |
| PHOTO-04 | 초기화 시 전체 사진의 임베딩을 클러스터링하여 인물별 그룹을 생성한다 |
| PHOTO-05 | 사용자가 선택한 그룹(들)의 평균 임베딩을 아이의 기준 벡터로 저장한다 |
| PHOTO-06 | 신규 사진은 기준 벡터와의 코사인 유사도로 자동 분류한다 |
| PHOTO-07 | 사진의 EXIF 날짜를 타임라인 날짜로 사용하며, EXIF 없을 시 파일 생성일을 사용한다 |
| PHOTO-08 | 모든 얼굴 인식 처리는 온디바이스에서 수행하며 외부 서버에 사진을 전송하지 않는다 |

## 4. 비기능 요구사항

| ID | 구분 | 요구사항 |
|---|---|---|
| NFR-01 | 성능 | 초기 사진 스캔은 백그라운드 스레드에서 수행하며, 초기화가 완료될 때까지 전체 화면 진행률 오버레이(처리된 사진 수 / 전체 사진 수, %)를 표시하고 모든 제스처 입력을 차단한다 |
| NFR-02 | 성능 | 타임라인 스크롤은 60fps 이상을 유지한다 |
| NFR-03 | 프라이버시 | 얼굴 인식은 온디바이스에서만 수행하며 외부 서버에 사진/임베딩을 전송하지 않는다 |
| NFR-04 | 프라이버시 | 카카오톡 파싱 데이터는 기기 내 Room DB에만 저장한다 |
| NFR-05 | 신뢰성 | 카카오톡 import 실패 시 이전 데이터는 유지되며 롤백된다 |
| NFR-06 | 호환성 | Android 10 (API 29) 이상을 지원한다 |
| NFR-07 | 저장소 | TFLite 모델 파일은 앱 번들에 포함하며 앱 크기는 100MB 이하를 목표로 한다 |
| NFR-08 | 디자인 | 모든 UI 컴포넌트, 모션, 색상 체계는 Material Design 3(M3) 가이드라인을 따른다 |

## 5. 데이터 모델

```
Child
├── id: UUID
├── name: String
├── birthDate: LocalDate
└── faceEmbeddings: List<FloatArray>  // 평균 임베딩 벡터(들)

Event
├── id: UUID
├── childId: UUID (FK)
├── date: LocalDate
├── category: Enum(SAID, DID, PHOTO, OTHER)
├── content: String
├── isFavorite: Boolean
├── source: Enum(KAKAO, PHOTO, MANUAL)
└── createdAt: Instant

PhotoRecord
├── id: UUID
├── eventId: UUID (FK)
├── localUri: String          // MediaStore URI
├── takenAt: Instant          // EXIF 날짜
├── faceEmbedding: FloatArray // 해당 사진에서 감지된 얼굴 임베딩
└── similarityScore: Float    // 기준 벡터 대비 유사도

KakaoRoom
├── id: UUID
├── roomName: String
└── lastImportedAt: Instant   // 증분 업데이트 기준점

KakaoMessage
├── id: UUID
├── roomId: UUID (FK)
├── sender: String
├── sentAt: Instant
├── content: String
└── contentHash: String       // 중복 방지용 SHA-256
```

## 7. 외부 연동 제약사항

| 연동 대상 | 방법 | 제약 |
|---|---|---|
| 카카오톡 | 내보내기 `.txt` 파일 파싱 | 공개 API 없음. 직접 채팅 읽기 불가 |
| 삼성 갤러리 인물 그룹 | ❌ 지원 불가 | 외부 앱 접근 API 없음 |
| Google Photos | ❌ 제한적 | 얼굴 그룹 API 2019년 이후 비공개 |
| Google Photos (사진 접근) | MediaStore 경유 | 기기에 다운로드된 사진만 접근 가능 |

### AI API 키 관리 정책

본 앱은 개인·가족용으로 제작되므로 아래 방식을 기본으로 채택한다.

| 방식 | 적용 시점 | 비고 |
|---|---|---|
| `local.properties` + `BuildConfig` (기본) | 개발 및 개인 배포 | APK에 키가 포함되므로 비공개 배포에 한함 |
| 백엔드 프록시 서버 | Play Store 공개 배포 시 전환 | 키가 서버에만 존재하여 노출 위험 없음 |

- API 키는 `local.properties`에 저장하고 `.gitignore`로 커밋에서 제외한다.
- `BuildConfig`를 통해 빌드 시 앱에 주입한다.
- Play Store 공개 배포를 고려할 경우, 키를 서버 측에서만 보유하는 프록시 API로 전환해야 한다.

## 8. 구현 우선순위 (Phase 계획)
