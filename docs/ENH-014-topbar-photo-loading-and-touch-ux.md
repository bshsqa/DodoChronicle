# ENH-014: 상단 액션, 신규 사진 로딩, 오버레이 터치 UX 정리

## 목표 (Objective)
#33, #27, #26을 하나의 UX 정리 작업으로 묶어, 타임라인 상단 액션과 수동 사진 로딩 흐름, 로딩/임포트 중 터치 처리를 일관되게 개선합니다.

이 작업의 핵심은 다음입니다.

- 상단 앱바는 자주 쓰는 핵심 액션만 남기고 복잡도를 낮춥니다.
- 카카오톡 임포트처럼 빈도 낮고 무거운 작업은 설정/옵션 메뉴로 이동합니다.
- `신규 사진 로딩`은 중복 스캔을 줄이고, 사용자가 실행 상태와 완료 후 확인 흐름을 명확히 알 수 있게 합니다.
- 수동 이벤트 추가와 사진 추가는 둘 다 "추가" 계열 액션으로 묶어, 사용 빈도와 화면 복잡도를 함께 고려합니다.
- 로딩/임포트/모달성 UI가 표시되는 동안 하위 버튼이 눌리지 않도록 터치 차단을 보장합니다.

---

## 관련 이슈
- #33 신규 사진 로딩 눌렀을 때 동작은?
- #27 터치 조절
- #26 상단 아이콘 순서 정렬

## 배경 (Background)
현재 상단 앱바에는 카카오톡 임포트, 즐겨찾기 필터, 설정, 검색, 사진 추가 등 여러 액션이 직접 노출되어 있습니다.

기능이 늘어나면서 다음 문제가 같이 생겼습니다.

- 설정 아이콘이 중간에 있어 일반적인 모바일 앱 기대와 다릅니다.
- 카카오톡 임포트는 무거운 작업인데 상단에 직접 노출되어 앱바가 복잡합니다.
- `신규 사진 로딩`을 수동으로 눌렀을 때 마지막 처리 시점 이후만 로딩하는지, 다시 50개를 반복 스캔하는지 흐름이 불명확합니다.
- 수동 사진 로딩은 백그라운드에서만 도는 것처럼 보여 사용자가 상태를 알기 어렵습니다.
- 로딩이 끝난 뒤 확인 필요한 사진이 있으면 확인/카테고리 선택 흐름이 바로 이어져야 하는데, 현재는 체감상 끊겨 보일 수 있습니다.
- 카카오톡 임포트나 로딩 UI가 떠 있어도 아래 아이콘이 터치되는 문제가 있습니다.

## 주요 요구사항 (Requirements)

### 1. 상단 앱바 액션 정리
상단 앱바는 타임라인에서 자주 쓰는 가벼운 액션 중심으로 정리합니다.

권장 방향:

- 상단 앱바 유지 후보:
  - 검색
  - 즐겨찾기 필터
  - 설정
- 설정 아이콘은 가장 오른쪽에 둡니다.
- 카카오톡 임포트는 상단 직접 버튼에서 제거하고 설정/옵션 메뉴 안으로 이동합니다.
- 기존 하단 `+` FAB는 "상단 앱바로 이동"한다는 의미가 아니라, 추가 계열 액션의 진입점으로 재검토합니다.
- 사진 추가는 수동 이벤트 추가보다 더 자주 쓰일 가능성이 있으므로, 둘을 분리해서 노출할지 하나의 추가 메뉴로 묶을지 결정합니다.

추가 액션 후보:

1. 하단 `+` FAB를 누르면 작은 액션 메뉴를 띄웁니다.
   - 사진 추가
   - 텍스트 이벤트 추가
2. 사진 추가를 기본 FAB로 두고, 텍스트 이벤트 추가는 보조 메뉴로 이동합니다.
3. 사진 추가와 텍스트 이벤트 추가를 둘 다 설정 메뉴로 넣습니다.

권장:

- 하단 `+` FAB는 유지합니다.
- `+`를 누르면 간단한 `DropdownMenu`를 띄워 `사진 추가`와 `텍스트 이벤트 추가`를 선택하게 합니다.
- 사진 추가가 더 자주 쓰인다면 리스트 첫 번째를 `사진 추가`로 둡니다.
- 설정 메뉴에는 무거운 관리/동기화 작업을 두고, 빠른 생성 액션은 FAB에 둡니다.
- 상단 앱바의 기존 `사진 추가` 버튼은 제거하고 `+` 메뉴 안으로 이동합니다.

상단 앱바 초기 구현 권장:

```text
[즐겨찾기] [검색] [설정]
```

설정 메뉴 안:

```text
카카오톡 대화 가져오기
신규 사진 로딩
문맥 업데이트
숨김 아이템
앱 데이터 초기화
```

### 2. 카카오톡 임포트 진입 경로 이동
카카오톡 임포트는 설정 메뉴 안으로 이동합니다.

- 기존 상단 카카오톡 버튼은 제거합니다.
- 설정 메뉴에서 `카카오톡 대화 가져오기`를 선택하면 기존 파일 선택 흐름을 실행합니다.
- 기존 `showKakaoMenu`, 파일 선택 launcher, import 실행 로직은 기능 유지하되 진입점만 바꿉니다.

### 2-1. 사진 로딩 경로 정리
현재 코드에는 사진을 다루는 경로가 크게 두 종류 있습니다.

1. `ScanForegroundService`
   - 초기 설정 또는 전체/최근 사진 얼굴 클러스터링 성격의 스캔입니다.
   - 갤러리에서 최근 사진들을 훑고 얼굴 embedding을 만들고 cluster를 구성합니다.
   - 현재 수동 `신규 사진 로딩` 버튼이 이 경로를 타면, "신규 사진만 로딩"이 아니라 "최근 N장 재스캔"처럼 동작할 수 있습니다.

2. `PhotoSyncWorker` / `SyncNewPhotosUseCase`
   - 이미 설정된 아이 얼굴 embedding과 새 사진을 비교해 자동 추가 또는 확인 필요 목록을 만듭니다.
   - 기존 저장 사진 URI를 제외하는 로직이 있습니다.
   - `pendingPhotos` 확인 UI와 자연스럽게 연결할 수 있습니다.

ENH-014의 `신규 사진 로딩`은 2번 경로로 정리합니다.

- 설정 메뉴의 `신규 사진 로딩`은 `ScanForegroundService`를 직접 실행하지 않습니다.
- `DATE_ADDED` 기준으로 새 후보 사진을 조회합니다.
- 조회 결과를 `SyncNewPhotosUseCase`에 넘겨 자동 추가/확인 필요 흐름을 실행합니다.
- 초기 설정이나 전체 재스캔 성격의 얼굴 클러스터링은 별도 기능으로 남깁니다.

### 3. 신규 사진 로딩 동작 정리
`신규 사진 로딩`은 이미 처리한 사진을 반복해서 다시 스캔하지 않아야 합니다.

정책:

- 마지막 사진 스캔 기준점을 저장합니다.
- 수동 `신규 사진 로딩`은 기준점 이후의 사진만 대상으로 합니다.
- 기준점이 없을 때만 초기 로딩 정책을 사용합니다.
- 초기 로딩이 현재 50개 제한이라면, 기준점 없는 최초 수동 스캔에서는 기존 정책을 유지합니다.
- 기준점 이후 사진이 0개면 사용자에게 완료 상태를 짧게 알려줍니다.

검토할 기준점 후보:

- `lastPhotoScanAt`: 마지막 스캔 실행 시각
- `lastProcessedPhotoTakenAt`: 마지막으로 처리한 사진의 촬영 시각
- `lastProcessedPhotoAddedAt`: 마지막으로 처리한 MediaStore 추가 시각

권장:

- 신규 사진 "발견" 기준은 `DATE_ADDED`를 사용합니다.
- 이벤트 날짜와 타임라인 배치는 기존처럼 `DATE_TAKEN`을 우선 사용합니다.
- `DATE_TAKEN`이 없거나 0이면 `DATE_ADDED * 1000`을 fallback으로 사용합니다.
- 마지막 스캔 기준점은 특정 파일/URI가 아니라 `lastProcessedPhotoAddedAtSeconds` 같은 시각 값으로 저장합니다.
- MediaStore timestamp 단위/지연 반영을 고려해 기준점 직전 일정 구간을 겹쳐서 다시 조회합니다.
- 사진 중복 방지는 기준 시각만 믿지 말고, 이미 저장된 `localUri`를 함께 제외합니다.

예시 정책:

```text
queryStart = max(0, lastProcessedPhotoAddedAtSeconds - overlapSeconds)
MediaStore.DATE_ADDED >= queryStart 인 사진 조회
이미 DB에 저장된 localUri 제외
처리 완료 후 조회된 사진들의 max(DATE_ADDED)를 lastProcessedPhotoAddedAtSeconds로 저장
```

이 방식은 다음 케이스를 더 안전하게 처리합니다.

- 오래전에 찍은 사진을 오늘 다운로드/복사해서 갤러리에 추가한 경우
- 사용자가 기준점이 된 특정 사진 파일을 삭제한 경우
- MediaStore 반영이 약간 늦게 들어온 경우

### 3-1. 이미 로딩된 원본 사진 삭제 대응
현재 앱은 사진 원본을 복사해 보관하지 않고 MediaStore URI를 저장합니다.

따라서 사용자가 갤러리에서 이미 로딩된 원본 사진을 삭제하면 다음 문제가 생길 수 있습니다.

- DB에는 사진 이벤트와 `PhotoRecord.localUri`가 남아 있습니다.
- 화면에서 Coil/ContentResolver가 해당 URI를 열지 못하면 썸네일 또는 전체화면 이미지가 비어 보일 수 있습니다.
- 얼굴 embedding은 DB에 남아 있으므로, 원본 사진이 삭제되어도 모델 업데이트에는 계속 사용될 수 있습니다.
- `getAllPhotoUris()` 중복 제외에도 남아 있으므로, 같은 URI가 다시 들어오는 특수 케이스에서는 재처리를 막을 수 있습니다.

ENH-014에서는 최소한 다음 정책을 함께 고려합니다.

- 신규 사진 로딩 또는 별도 정리 시 저장된 `localUri`가 실제로 열리는지 확인합니다.
- 열 수 없는 사진은 `missing` 상태로 표시하거나, 사진 이벤트/PhotoRecord를 정리합니다.
- 원본이 사라진 사진의 embedding은 모델 업데이트 대상에서 제외합니다.
- 사용자에게 자동 삭제 대신 `원본 사진을 찾을 수 없음` 상태를 보여주고 정리할 수 있게 하는 방식을 우선 검토합니다.

향후 DB 필드 후보:

```kotlin
val mediaStoreAddedAt: Long = 0L
val lastSeenAt: Long = 0L
val isMissing: Boolean = false
```

### 4. 신규 사진 로딩 UI
수동으로 `신규 사진 로딩`을 눌렀을 때 사용자는 실행 중임을 즉시 알아야 합니다.

요구사항:

- 로딩 시작 시 화면에 진행 UI를 표시합니다.
- 로딩 중 설정 메뉴는 닫습니다.
- 중복 실행을 막습니다.
- 완료 시 결과를 알려줍니다.

표시 예시:

```text
신규 사진 분석 중...
```

완료 후:

- 확인 필요한 사진이 있으면 `PendingPhotosDialog` 또는 기존 확인 UI를 바로 표시합니다.
- 확인 필요한 사진이 없으면 Snackbar 또는 짧은 완료 메시지를 표시합니다.

### 5. 완료 후 확인 흐름
`신규 사진 로딩` 완료 후 확인 필요한 사진이 생겼다면 사용자가 바로 처리할 수 있어야 합니다.

요구사항:

- `pendingPhotos`가 새로 생긴 경우 자동으로 확인 다이얼로그를 엽니다.
- 기존 `PendingPhotosBanner`는 유지하되, 수동 로딩 직후에는 한 번 바로 열어줍니다.
- 사용자가 닫으면 이후에는 배너를 통해 다시 들어갈 수 있습니다.

### 5-1. 자동/수동 사진 로딩 UX 차이
자동 사진 로딩과 수동 사진 로딩은 같은 사진 판정 로직을 사용하되, 사용자 방해 수준을 다르게 둡니다.

#### 6시간 자동 사진 로딩
- 유사도 높음: 자동으로 사진 이벤트를 추가합니다.
- 유사도 낮음: 조용히 제외합니다.
- 애매함: pending 사진 목록에 저장합니다.
- 자동 실행 중에는 다이얼로그를 자동으로 띄우지 않습니다.
- 앱이 foreground에 있다면 Snackbar로 자동 추가 결과를 가볍게 알립니다.

예시:

```text
사진 4장을 자동 추가했습니다.
```

- Snackbar가 사라져도 pending 사진은 사라지지 않아야 합니다.
- `PendingPhotosBanner` 또는 설정 메뉴 항목을 통해 나중에 다시 들어갈 수 있어야 합니다.
- Snackbar와 Banner가 동시에 같은 메시지를 반복하지 않도록 역할을 분리합니다.
  - Snackbar: 방금 자동 실행의 결과를 잠깐 알립니다.
  - Banner: 아직 처리하지 않은 pending 사진이 남아 있음을 계속 보여주고 선택창 진입점 역할을 합니다.
- 자동 추가 없이 pending만 새로 생긴 경우에는 Snackbar를 생략하거나 `확인 필요한 사진이 생겼습니다`를 1회만 표시합니다.
- pending 처리 진입은 기본적으로 Banner가 담당합니다.

#### 수동 `신규 사진 로딩`
- 유사도 높음: 자동으로 사진 이벤트를 추가합니다.
- 유사도 낮음: 조용히 제외합니다.
- 애매함: pending 사진 목록에 저장하고, 로딩 완료 직후 확인 다이얼로그를 바로 엽니다.
- 자동 추가만 있고 애매한 사진이 없으면 Snackbar로 결과만 알립니다.
- 새 후보가 없으면 Snackbar로 `새로 추가할 사진이 없습니다`를 표시합니다.

예시:

```text
사진 5장을 자동 추가했습니다.
```

또는:

```text
사진 5장을 자동 추가했고, 확인 필요한 사진 3장이 있습니다.
```

### 5-2. Pending 사진 저장 정책
현재 `TimelineUiState.pendingPhotos`는 화면 상태이므로, 앱 프로세스가 종료되거나 6시간 자동 worker에서 생긴 pending 후보를 장기 보관하기에는 부족합니다.

ENH-014에서는 pending 후보를 영속 저장할 수 있는 구조를 추가합니다.

후보 테이블:

```kotlin
data class PendingPhotoEntity(
    val uri: String,
    val takenAt: Long,
    val addedAtSeconds: Long,
    val similarity: Float,
    val faceEmbeddingJson: String,
    val createdAt: Long,
    val source: PendingPhotoSource
)
```

정책:

- pending 후보는 `uri` 기준으로 unique 처리합니다.
- 자동 로딩에서 애매한 사진이 나오면 기존 pending에 append합니다.
- 이미 pending에 있는 URI는 다음 자동/수동 로딩에서 재처리하지 않습니다.
- 사용자가 승인하면 사진 이벤트/PhotoRecord를 만들고 pending에서 제거합니다.
- 사용자가 거절하면 pending에서 제거합니다.
- Snackbar는 일시적 알림일 뿐이며, pending 후보의 생명주기와 연결하지 않습니다.
- pending이 하나라도 남아 있으면 `PendingPhotosBanner`를 계속 표시합니다.
- 배너를 닫는 UI를 만들더라도, 설정 메뉴에서 `확인 필요한 사진`으로 다시 진입할 수 있어야 합니다.

### 6. 오버레이/모달 표시 중 하위 터치 차단
카카오톡 임포트, 사진 로딩, 문맥 업데이트, 전체 화면 로딩 UI가 표시되는 동안 하위 앱바/버튼이 터치되지 않아야 합니다.

구현 원칙:

- 전체 화면 또는 상단 오버레이가 떠 있을 때는 투명 터치 차단 레이어를 둡니다.
- 단순 `Box` 배경만 올리지 말고 pointer input/clickable 등으로 터치를 소비합니다.
- 모달성 진행 UI는 가능한 `Dialog`, `AlertDialog`, 또는 full-screen blocking Box로 구현합니다.

Compose 예시:

```kotlin
Box(
    modifier = Modifier
        .fillMaxSize()
        .background(Color.Black.copy(alpha = 0.25f))
        .pointerInput(Unit) {
            awaitPointerEventScope {
                while (true) {
                    awaitPointerEvent()
                }
            }
        }
)
```

또는 접근성/동작이 더 자연스러운 경우:

```kotlin
Box(
    modifier = Modifier
        .fillMaxSize()
        .clickable(
            indication = null,
            interactionSource = remember { MutableInteractionSource() }
        ) {}
)
```

## 구현 설계

### 1. UI 상태 점검
`TimelineUiState`에서 다음 상태를 확인하거나 추가합니다.

- 사진 스캔 실행 중 여부
- 사진 스캔 완료 결과
- 수동 사진 스캔 직후 pending dialog 자동 표시 여부
- 임포트/문맥 업데이트 등 blocking 작업 여부

후보:

```kotlin
val isBlockingOverlayVisible: Boolean
val shouldOpenPendingPhotosAfterScan: Boolean = false
```

단, 파생 가능한 상태라면 별도 필드를 늘리지 않고 화면에서 계산합니다.

### 2. 상단 앱바 정리
`TimelineScreen`의 `TopAppBar.actions`를 재배치합니다.

초기 권장 순서:

1. 즐겨찾기 필터
2. 검색
3. 설정

카카오톡 임포트는 `SettingsMenuDialog`로 이동합니다.

### 3. 설정 메뉴 확장
`SettingsMenuDialog`에 `카카오톡 대화 가져오기` 항목을 추가합니다.

- 클릭 시 메뉴를 닫고 기존 카카오톡 파일 선택 흐름을 실행합니다.
- 기존 pending retry, 신규 사진 로딩, 문맥 업데이트, 숨김 아이템, 초기화 항목과 시각적으로 구분합니다.

### 4. 신규 사진 스캔 기준점 저장
사진 스캔 use case와 repository 흐름을 확인합니다.

기준점은 DataStore에 저장합니다.

```kotlin
AppPrefsKeys.LAST_PHOTO_SYNC_ADDED_AT_SECONDS
```

스캔 쿼리는 기준점 이후 사진만 조회하고, 이미 저장된 사진 URI는 제외합니다.

구현 정책:

- `MediaStore.Images.Media.DATE_ADDED`를 projection에 포함합니다.
- query 기준은 `DATE_ADDED >= max(0, LAST_PHOTO_SYNC_ADDED_AT_SECONDS - 86400)`으로 둡니다.
- 24시간 overlap을 두어 MediaStore 반영 지연과 경계값 누락을 줄입니다.
- 조회된 사진 중 이미 DB에 저장된 `localUri`는 `SyncNewPhotosUseCase`에서 다시 제외합니다.
- 처리 성공 후, 조회 후보 중 가장 큰 `DATE_ADDED` 값을 `LAST_PHOTO_SYNC_ADDED_AT_SECONDS`에 저장합니다.
- `DATE_ADDED`가 없거나 0인 사진은 후보에서 제외하거나 `DATE_TAKEN` 기반 fallback을 적용하되, 기준점 갱신에는 사용하지 않습니다.

### 5. 사진 로딩 완료 후 pending dialog 자동 표시
수동 스캔이 끝났고 `pendingPhotos.isNotEmpty()`이면 `showPendingDialog = true`가 되도록 연결합니다.

주의:

- 앱 시작 시 기존 pending 사진이 있다고 매번 자동으로 다이얼로그를 띄우지는 않습니다.
- 수동 `신규 사진 로딩` 직후에만 자동 표시합니다.

### 6. 터치 차단 레이어 공통화
현재 import/progress overlay들이 실제로 터치를 소비하는지 확인합니다.

필요 시 공통 composable을 둡니다.

```kotlin
@Composable
private fun BlockingOverlay(content: @Composable BoxScope.() -> Unit)
```

적용 후보:

- 카카오톡 대화 분석 중
- 신규 사진 분석 중
- 문맥 업데이트 중
- import 완료 overlay가 modal이어야 하는 경우

## 수용 기준 (Acceptance Criteria)

### #26 기준
- 설정 아이콘은 상단 앱바 가장 오른쪽에 표시된다.
- 카카오톡 임포트는 상단 앱바에서 제거되고 설정 메뉴 안에서 실행된다.
- 상단 앱바는 핵심 액션 중심으로 단순해진다.

### #33 기준
- `신규 사진 로딩`은 마지막 처리 기준 이후 사진만 대상으로 한다.
- 같은 사진 50개를 반복해서 다시 로딩하지 않는다.
- 수동 사진 로딩 중에는 사용자가 진행 상태를 볼 수 있다.
- 로딩 완료 후 확인 필요한 사진이 있으면 확인 UI가 바로 열린다.
- 확인 필요한 사진이 없으면 완료 상태를 알 수 있다.

### #27 기준
- 카카오톡 임포트/사진 로딩/문맥 업데이트 등 오버레이 표시 중 하위 아이콘이 눌리지 않는다.
- 모달성 UI 바깥의 터치가 하위 화면으로 전달되지 않는다.
- 중복 실행이 필요한 작업이 아니라면 로딩 중 같은 작업을 다시 누를 수 없다.

## 리스크 (Risks)

- MediaStore의 `DATE_TAKEN`과 `DATE_ADDED` 값은 기기/사진 출처에 따라 비어 있거나 부정확할 수 있습니다.
- 기준점 기반 신규 스캔만 사용하면 과거에 추가된 사진을 놓칠 수 있으므로, 이미 저장된 URI 제외와 적절한 fallback이 필요합니다.
- 카카오톡 임포트를 설정 메뉴로 이동하면 발견성이 낮아질 수 있습니다. 메뉴 문구를 명확하게 둬야 합니다.
- Pending dialog 자동 표시가 너무 공격적이면 사용자를 방해할 수 있으므로 수동 스캔 직후로 제한합니다.
