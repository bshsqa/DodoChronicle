# ENH-013: Gemini JSON 응답 복구와 필드 정규화

## 목표 (Objective)
카카오톡 임포트 중 Gemini 응답이 부분적으로 깨지더라도, 살릴 수 있는 이벤트는 최대한 보존합니다.

ENH-012에서 Gemini 응답 스키마가 확장되면서 이벤트 필수 필드와 검색 보조 필드가 함께 반환됩니다. 대부분은 정상 JSON으로 오겠지만, 응답 전체 또는 일부 필드가 기대 스키마와 다를 수 있습니다. 이때 chunk 전체를 곧바로 실패 처리하지 않고, 합리적인 복구 단계를 거칩니다.

---

## 실패 유형

### 1. JSON 전체 파싱 실패
Gemini가 chunk 하나에 대해 응답을 줬지만, 응답 전체를 JSON 배열로 읽을 수 없는 경우입니다.

예:

- JSON 앞뒤에 설명 문장이 붙음
- JSON 배열이 중간에서 끊김
- 우리가 기대한 배열이 아니라 wrapper 객체로 옴
- 쉼표, 따옴표, 코드펜스 등이 깨짐

이 경우는 이벤트별 판단이 불가능하므로 **청크 응답 전체 이슈**로 취급합니다.

처리:

- 원본 카카오 chunk와 이전 Gemini 응답 전체를 함께 보내 repair 요청을 1회 수행합니다.
- repair 성공 시 복구된 이벤트를 사용합니다.
- repair 실패 시 기존처럼 해당 chunk를 실패 처리하고 retry chunk로 남깁니다.

### 2. 이벤트 필수 필드 파싱 실패
JSON 배열 자체는 읽혔지만, 일부 이벤트의 필수 필드가 깨진 경우입니다.

필수 필드:

- `date`
- `category`
- `content`

처리:

- 정상 이벤트는 그대로 사용합니다.
- 깨진 이벤트들은 한 번에 모아, 원본 카카오 chunk와 함께 repair 요청을 1회 수행합니다.
- 복구된 이벤트만 추가로 사용합니다.
- repair로도 복구되지 않은 이벤트는 버립니다.

### 3. 검색 보조 필드 파싱 실패
이벤트 필수 필드는 정상이나 검색 보조 필드가 누락되거나 잘못된 타입으로 온 경우입니다.

검색 보조 필드:

- `searchSummary`
- `searchTags`
- `searchAliases`
- `relatedKeywords`

처리:

- repair 요청을 보내지 않습니다.
- 로컬에서 가능한 만큼 정규화합니다.
- 그래도 사용할 수 없으면 fallback을 적용합니다.

예:

```json
{
  "searchTags": "병원, 열, 진료",
  "relatedKeywords": [123, "걱정", null]
}
```

정규화 결과:

```json
{
  "searchTags": ["병원", "열", "진료"],
  "relatedKeywords": ["걱정"]
}
```

## Repair 요청 정책

- chunk당 repair 요청은 최대 1회만 수행합니다.
- 전체 JSON 파싱 실패와 필수 필드 실패가 동시에 발생할 수는 없습니다. 전체 JSON 파싱에 실패하면 이벤트별 분석이 불가능하기 때문입니다.
- 검색 보조 필드 문제만으로는 repair 요청을 보내지 않습니다.
- repair 요청도 실패하면 해당 chunk는 기존 실패 처리 흐름을 따릅니다.

repair 요청에는 다음 정보를 포함합니다.

- 원본 카카오 chunk
- 이전 Gemini 응답
- 실패 유형
- 문제가 된 이벤트의 raw JSON 또는 문제 설명
- 반드시 스키마에 맞는 JSON 배열만 반환하라는 지시

---

## 수용 기준 (Acceptance Criteria)

- 전체 JSON 파싱 실패 시 repair 요청을 1회 수행한다.
- repair 성공 시 chunk를 실패 처리하지 않고 이벤트를 저장한다.
- JSON 배열은 정상이나 일부 이벤트 필수 필드가 깨진 경우, 정상 이벤트는 살리고 깨진 이벤트만 repair한다.
- 검색 보조 필드만 깨진 경우 Gemini 재요청 없이 로컬 정규화/fallback으로 처리한다.
- repair는 chunk당 최대 1회만 수행한다.
- repair 실패 시 기존 retry chunk 흐름이 유지된다.
