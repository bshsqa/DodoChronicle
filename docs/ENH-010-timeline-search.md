# ENH-010: 타임라인 통합 검색 기능 (Timeline Search)

## 목표 (Objective)
타임라인 내비게이션을 방해하지 않는 깔끔한 다이얼로그 형태의 검색 UI를 제공하고, 단순 키워드 매칭과 텍스트 임베딩을 활용한 Vector Similarity(벡터 유사도) 기반의 의미 기반 문맥 검색을 모두 지원합니다.

## 주요 요구사항 (Requirements)

### 1. 검색 다이얼로그 UI
- `TimelineScreen`의 최상단 툴바(TopAppBar) 우측에 돋보기(검색) 아이콘을 배치합니다.
- 아이콘 클릭 시 타임라인 상단에 조그마한 검색 다이얼로그(혹은 Dropdown/SearchBar 형태의 오버레이)가 나타납니다.
- **다이얼로그 구성 요소:**
  - 검색어 입력창 (TextField)
  - 검색 실행 버튼
  - "문맥 포함 (의미 기반 검색)" 체크박스 또는 토글 버튼

### 2. 검색 결과 정렬 정책 유지
- 검색 결과의 연관성(유사도 점수)이 높든 낮든, 최종적으로 화면에 보여지는 정렬 순서는 **항상 기존의 날짜순 정렬 정책을 100% 유지**합니다. 
- 시간의 흐름(Chronicle)을 해치지 않고, 검색된 이벤트들을 타임라인에 필터링해 줍니다.

### 3. 두 가지 검색 모드의 명확한 구분

#### A. 일반 키워드 검색 ("문맥 포함" 해제 시)
- 띄어쓰기를 기준으로 단어를 분리하여, **입력한 모든 단어가 포함되어 있는지 (AND 조건)** 검사합니다.
- **검색 범위:** 한 줄 요약(`content`), 상세 내용(`longContent`), 원본 대화 발췌(`rawExcerpt`) 모두 포함.
- (예: "울산 집" 검색 시 텍스트 내에 "울산"과 "집" 두 단어가 모두 있어야 노출되며, 순서는 무관합니다.)
- 고급 검색: 만약 쌍따옴표(`"울산 집"`)로 묶어서 검색할 경우, 띄어쓰기까지 포함해 정확히 일치하는 문구만 찾아냅니다.

#### B. 의미 기반 문맥 검색 ("문맥 포함" 체크 시)
- 텍스트 임베딩(Vector)과 코사인 유사도를 활용하여 **의미적 연관성**을 기준으로 검색을 수행합니다. (별도의 결과 생성(Generation)이나 Top-N 추출이 아닌, 유사도 임계치를 넘는 모든 결과를 필터링하여 보여줍니다.)
- (예: "아이가 아팠던 때" 검색 시, 텍스트에 "아팠던"이 없더라도 "열이 펄펄 끓네", "응급실 갔다왔어" 등의 문맥적 의미가 유사한 이벤트를 모두 찾아냅니다.)

#### C. 검색 제외 대상
- **사진 이벤트(`EventCategory.PHOTO`):** 현재 사진 이벤트에는 텍스트 정보나 문맥이 존재하지 않으므로, 키워드 검색 및 문맥 검색의 대상에서 모두 제외됩니다.

## 구현 설계 (Implementation Details)

### 1. Text Embedding 데이터 구조 추가
의미 기반 검색을 위해 모든 텍스트 이벤트에 대한 임베딩 벡터가 필요합니다.
*   **Database Schema (`EventEntity.kt`):** 
    *   `textEmbeddingJson: String = "[]"` 필드를 추가하여 텍스트 임베딩 벡터(`List<Float>`)를 저장합니다.
    *   DB Migration이 필요합니다.
*   **On-Device Embedding 연동 (`TextEmbeddingEngine.kt`):**
    *   Gemini API 호출에 따른 네트워크 지연과 비용을 없애기 위해, 온디바이스 텍스트 임베딩을 수행합니다.
    *   MediaPipe Text Embedder 혹은 TFLite 기반 경량 모델(예: Universal Sentence Encoder Lite 등)을 `assets` 폴더에 탑재하여, 네트워크 없이 로컬에서 벡터 값을 반환하는 로직을 구현합니다.

### 2. 이벤트 등록 및 마이그레이션 로직
*   **신규 이벤트:** 카카오 대화 분석으로 이벤트가 추출될 때, 해당 이벤트의 내용(`content` + `longContent`)을 결합하여 임베딩 벡터를 생성한 후 DB에 함께 저장합니다.
*   **기존 이벤트 동기화 (불필요):** 현재 개발 단계이므로 기존 레거시 데이터를 위한 임베딩 일괄 갱신(Migration/Backfill) 로직은 구현하지 않습니다. 출시 후 모든 사용자는 초기화 단계를 거치므로 임베딩 값이 없는 텍스트 이벤트는 존재하지 않는다고 가정합니다.

### 3. 카카오톡 대화 전처리 (Preprocessing)
의미 기반 검색의 정확도를 높이고 AI(Gemini)의 혼란을 막기 위해, 이벤트 추출 전 카카오톡 대화 내역에서 불필요한 시스템 미디어 메시지를 필터링합니다.
*   **필터링 대상 패턴 예시:** `<사진 읽지 않음>`, `<동영상 읽지 않음>`, `사진`, `사진 N장`, `동영상`, 실제 파일명(예: `IMG_1234.jpg`) 등
*   **삭제 방식:** 내용만 지우고 "날짜, 발신자 : " 부분만 남겨두는 것이 아니라, **해당 메시지 라인(객체) 전체를 리스트에서 완전히 제거(Drop)**합니다.
*   이러한 시스템성 메시지 전체를 사전에 삭제(전처리)함으로써, 추출된 텍스트와 임베딩 벡터에 무의미한 노이즈가 섞이는 것을 원천 차단합니다.

### 4. 검색 실행 로직 (`TimelineViewModel`)
*   **키워드 검색 처리:** Room DB의 `LIKE` 쿼리를 조합하여 AND 조건을 구현합니다. (예: `content LIKE '%단어1%' AND content LIKE '%단어2%'`)
*   **문맥 검색 처리:**
    1. 사용자가 검색 버튼을 누르면, 검색어의 임베딩 벡터를 로컬 `TextEmbeddingEngine`을 통해 즉시 추출합니다.
    2. DB에서 모든 텍스트 이벤트를 메모리로 불러오거나 Flow로 받아옵니다. (현재 로컬 환경 상 HNSW 인덱싱이 없으므로 메모리에서 Cosine Similarity 계산)
    3. 각 이벤트의 `textEmbedding`과 검색어 벡터 간의 코사인 유사도(Cosine Similarity)를 계산합니다.
    4. 유사도 임계치(Threshold, 예: 0.65 이상)를 넘는 이벤트만 필터링합니다.
*   **결과 반영:** 필터링된 이벤트의 ID 목록을 타임라인 리스트에 반영하여 화면을 갱신합니다. (정렬은 날짜순 유지)

## User Review Required

> [!TIP]
> 온디바이스 모델을 사용하므로 검색 시 API 호출 딜레이나 비용, Rate Limit 걱정이 전혀 없습니다! 오프라인 상태에서도 로컬에서 매우 빠르고 안전하게 의미 기반(Vector) 문맥 검색이 가능해집니다.
